# Milestone 2 Design

## Scope

Milestone 2 replaces the fixture-backed `03_analysis/` evidence layer with a live analysis pipeline that
ingests real media, generates analysis artifacts, and serves the same downstream contract that Milestone 1
already proved with fixtures.

The milestone boundary is:

- in scope:
  - live generation of `03_analysis/assets.json`
  - live generation and enrichment of `03_analysis/segments.json`
  - live generation of `03_analysis/transcripts/TR_*.json`
  - generated contact sheets, posters, filmstrips, and waveform derivatives that back `media-mcp`
  - `analysis_policy.yaml` and repo default policy
  - wiring `media-mcp` from fixture repository mode to live repository mode without changing tool interfaces
  - validation and tests that prove the M1 compiler loop can consume live-produced analysis artifacts
- out of scope:
  - embeddings-based semantic search
  - product-plane agent prompt changes
  - timeline compiler changes beyond consuming the same artifact shapes
  - OTIO round-trip changes
  - final render / packaging improvements

Milestone 2 must preserve the architecture gates in [ARCHITECTURE.md](/Users/mocchalera/Dev/video-os-v2-spec/ARCHITECTURE.md) and the `media-mcp` tool interface in [contracts/media-mcp.md](/Users/mocchalera/Dev/video-os-v2-spec/contracts/media-mcp.md).

### Success Criteria

Milestone 2 is complete when all of the following are true:

1. A project with real source media can run `ingest -> segment -> contact_sheet -> stt -> vlm` and produce
   schema-valid `assets.json`, `segments.json`, and `transcripts/*.json`.
2. The same source inputs, same schema-valid merged `analysis_policy`, and same canonical request tuple
   (`connector_version`, `model_snapshot`, `prompt_hash`, `response_format`, `chunking_strategy`,
   `ffmpeg_version`) produce byte-stable JSON artifacts and deterministic derivative paths on re-run.
3. Existing `media-mcp` tools can answer against the live artifacts without interface changes.
4. A Milestone 1 editorial loop can run against a project whose `03_analysis/` directory was generated by the
   live pipeline rather than hand-authored fixtures.
5. Partial failures do not destroy already-computed artifacts; they yield a machine-readable gap report and a
   `qc_status` of `partial` or `blocked` (depending on severity) instead of silently succeeding.

### Test Gates

The success criteria above are accepted by the following tests:

- unit:
  - deterministic ID assignment for assets, segments, transcripts, and contact sheets
  - ffmpeg/ffprobe output parsing to artifact fields
  - deterministic reducer merge for shard outputs and atomic rename discipline
  - STT transcript alignment into segment excerpts
  - STT chunk merge and speaker label normalization across chunk boundaries
  - VLM enrichment merge logic and cost-budget enforcement
  - policy resolution: `runtime/analysis-defaults.yaml` + project override + schema validation
- integration:
  - bundled short media fixture produces stable `assets.json`, `segments.json`, contact sheets, and waveforms
  - mocked OpenAI/Gemini responses produce stable transcript and enrichment artifacts
  - provider failure injection produces stable `gap_report.yaml` and `analysis_gaps` projection
- e2e:
  - live-generated `03_analysis/` + hand-authored test `04_plan/` artifacts compile through the M1 loop
  - `media.project_summary`, `media.list_assets`, `media.peek_segment`, and `media.read_transcript_span`
    succeed against live artifacts
- manual smoke:
  - `rokutaro-bicycle` and `AX-1 D4887` complete analysis with either `qc_status: ready` or `qc_status: partial`
    plus explicit gaps

## M1 To M2 Connection

Milestone 1 proved the editorial loop with fixture data under `projects/sample/03_analysis/`. Milestone 2
keeps the same downstream shape, but replaces the origin of each artifact:

| M1 fixture artifact | M2 live producer | Notes |
| --- | --- | --- |
| `03_analysis/assets.json` | ffprobe ingest connector | same root shape, enriched with stream metadata |
| `03_analysis/segments.json` | ffmpeg scene detection + Gemini enrichment + STT excerpt alignment | same core fields, richer optional fields |
| `03_analysis/transcripts/TR_*.json` | OpenAI STT connector | same transcript root path pattern |
| hand-authored summaries/tags | Gemini enrichment | now model-generated with provenance |
| hand-authored quality flags | ffmpeg heuristics + Gemini QA | merged into one normalized flag vocabulary |

Downstream impact:

- `selects_candidates.yaml` still references `segment_id`, `asset_id`, `src_in_us`, and `src_out_us`.
- the compiler still consumes `selects_candidates.yaml` rather than raw `03_analysis` artifacts.
- `media-mcp` stays the AI-facing evidence layer; M2 changes the backing repository, not the contract.

## M2 To M3 Handoff

Milestone 3 agents should consume M2 output through two surfaces:

- raw artifacts under `projects/*/03_analysis/`
- unchanged `media-mcp` tools backed by those artifacts

The primary M3 entrypoint is the `footage-triager` role in
[agent-src/roles/footage-triager.yaml](/Users/mocchalera/Dev/video-os-v2-spec/agent-src/roles/footage-triager.yaml),
which already expects:

- `projects/*/03_analysis/*`
- transcripts
- contact sheets
- quality-backed segment evidence

M2 therefore owns the `intent_locked -> media_analyzed` transition in the architecture state machine.
M3 starts only after `03_analysis/` is materialized and `media.project_summary` can report a usable analysis
status.

Gate rule for handoff:

- automated M3 entry requires `qc_status: ready`
- `qc_status: partial` is valid for debugging, manual inspection, and smoke verification, but it does not
  satisfy the `intent_locked -> media_analyzed` gate

## Analysis Architecture

The live pipeline keeps the repository layering from [ARCHITECTURE.md](/Users/mocchalera/Dev/video-os-v2-spec/ARCHITECTURE.md):

```text
source media
  -> ffmpeg / ffprobe engine
  -> OpenAI STT connector
  -> Gemini video understanding connector
  -> 03_analysis artifacts
  -> media-mcp repository
  -> M1 compiler loop / M3 agents
```

Important design decisions:

- ffmpeg/ffprobe remain deterministic engines, not provider connectors.
- Gemini and OpenAI remain provider-specific adapters behind provider-agnostic TypeScript interfaces.
- `timeline.json` remains untouched by M2; this milestone only makes `03_analysis/` live.
- time is canonicalized as `src_in_us` / `src_out_us` everywhere.

### Non-Functional Requirements

Milestone 2 adds external APIs and large local binaries, so its non-functional requirements must be explicit:

- determinism
  - same canonical request tuple => same JSON outputs, even without cache reuse
- reliability
  - stage retries are bounded and surfaced in `gap_report.yaml`; no silent fallback to empty semantics
- cost control
  - Gemini and OpenAI concurrency and per-segment budgets come from policy, not ad hoc code constants
- security
  - API credentials live only in runtime environment variables or local secret stores
  - provider request IDs may be stored in provenance, but raw auth material and raw SDK payload dumps may not
    be persisted in artifacts
- portability
  - canonical artifacts must not depend on machine-specific absolute source paths
  - local absolute paths may appear only in a non-canonical run manifest

### Operational Artifacts

In addition to the canonical analysis outputs, M2 introduces two operational artifacts:

- `03_analysis/gap_report.yaml`
  - partial failure ledger used by MCP status and retries
- `03_analysis/run_manifest.json`
  - non-canonical execution metadata for reproducibility and debugging
  - stores:
    - source locator to local absolute path map
    - connector versions
    - ffmpeg version
    - model aliases and resolved model snapshots
    - provider endpoints and response formats
    - prompt template hashes
    - chunking strategies and chunk-boundary hashes
    - stage-specific request hashes
    - policy hash
    - stage timings
    - started / completed timestamps

`run_manifest.json` is intentionally excluded from agent-facing contracts. It exists for operators, tests, and
postmortem debugging.

## Connector Design

### 1. ffmpeg / ffprobe Connector

Responsibilities:

- inspect source media with `ffprobe`
- assign deterministic `asset_id`
- detect shot boundaries and emit base `segments.json`
- compute deterministic quality heuristics that do not require an LLM
- generate contact sheets
- generate posters
- generate segment filmstrips
- generate waveform masters

Input:

- absolute or project-relative source file path (`.mov`, `.mp4`, `.mxf`, `.wav` where relevant)

Primary outputs:

- `03_analysis/assets.json`
- `03_analysis/segments.json` with base segment fields
- `03_analysis/contact_sheets/*.png`
- `03_analysis/contact_sheets/*.json`
- `03_analysis/posters/*.jpg`
- `03_analysis/filmstrips/*.png`
- `03_analysis/waveforms/*.png`

#### ID Rules

Milestone 2 chooses hash-stable asset IDs instead of discovery-order IDs.

- `source_fingerprint`
  - computed from `sha1(first_16mb + file_size + duration_us + normalized_stream_signature)`
  - path is intentionally excluded so file moves do not change identity
- `asset_id`
  - `AST_<source_fingerprint[0:8].upper()>`
  - collision handling extends the hash suffix before any fallback numbering is used
- `segment_id`
  - `SEG_<asset_id>_<segment_ordinal_4>`
  - `segment_ordinal` is assigned after sorting boundaries by `src_in_us`
- `contact_sheet_id`
  - `CS_<asset_id>_<page_2>`
- `transcript_ref`
  - `TR_<asset_id>`

Trade-off:

- sequential IDs like `AST_001` are easier to read, but they are not stable when files are added or reordered
- hash-stable asset IDs are less pretty, but they are the safer choice for re-runs, gap recovery, and later
  round-trip references

The fixture format remains valid because schemas will accept both old fixture IDs and the new live IDs.

#### Metadata Extraction

`assets.json` is seeded from `ffprobe -show_format -show_streams -print_format json`.

Fields derived deterministically:

| `assets.json` field | Source | Notes |
| --- | --- | --- |
| `asset_id` | connector ID rule | stable per source fingerprint |
| `filename` | normalized basename | fixture-compatible display field |
| `duration_us` | `format.duration` or primary stream duration | canonicalized to integer microseconds |
| `video_stream.width` / `height` | video stream metadata | required for live profile |
| `video_stream.fps_num` / `fps_den` | normalized `avg_frame_rate` | reduced fraction |
| `video_stream.codec` | `codec_name` | string |
| `audio_stream.sample_rate` | audio stream metadata | omitted if no audio stream |
| `audio_stream.channels` | audio stream metadata | omitted if no audio stream |
| `audio_stream.codec` | `codec_name` | string |
| `role_guess` | derived after STT/VLM enrichment | dominant evidence: `interview`, `b-roll`, `texture`, `hybrid`, or `unknown` |
| `has_transcript` | false at ingest, updated after STT | deterministic stage update |
| `transcript_ref` | `TR_<asset_id>` if audio present, else `null` | stable path key |
| `segments` / `segment_ids` | scene detection result | filled after segmentation |
| `contact_sheet_ids` | contact sheet generation result | one or more page ids |
| `poster_path` | poster derivative reducer | deterministic asset poster path |
| `waveform_path` | waveform master path | asset-level master waveform |

Path rule:

- `assets.json` may store a project-relative source locator only when the source lives under the project root
- machine-local absolute paths stay in `run_manifest.json`, not in canonical artifacts

#### Shot Boundary Detection

Milestone 2 chooses classic ffmpeg scene detection for the first live implementation.

Canonical detector command set:

- boundary candidates
  - `ffmpeg -i <src> -vf "select='gt(scene,${quality_thresholds.scene_threshold})',showinfo" -an -f null -`
- black coverage
  - `ffmpeg -i <src> -vf "blackdetect=d=${quality_thresholds.blackdetect_duration_s}:pix_th=${quality_thresholds.blackdetect_pix_th}:pic_th=${quality_thresholds.blackdetect_pic_th}" -an -f null -`
- frozen frames
  - `ffmpeg -i <src> -vf "freezedetect=n=${quality_thresholds.freezedetect_noise_db}dB:d=${quality_thresholds.freezedetect_duration_s}" -an -f null -`
- silence windows
  - `ffmpeg -i <src> -af "silencedetect=n=${quality_thresholds.silencedetect_noise_db}dB:d=${quality_thresholds.silencedetect_duration_s}" -vn -f null -`

Canonical defaults in `analysis_policy.yaml`:

- `quality_thresholds.scene_threshold: 0.30`
- `quality_thresholds.min_segment_duration_us: 750000`
- `quality_thresholds.merge_gap_us: 200000`
- `quality_thresholds.blackdetect_pic_th: 0.98`
- `quality_thresholds.blackdetect_pix_th: 0.10`
- `quality_thresholds.blackdetect_duration_s: 0.15`
- `quality_thresholds.silencedetect_noise_db: -35`
- `quality_thresholds.silencedetect_duration_s: 0.35`
- `quality_thresholds.freezedetect_noise_db: -50`
- `quality_thresholds.freezedetect_duration_s: 0.50`

Boundary assembly rules:

- collect cut candidates from `showinfo` PTS timestamps
- merge adjacent cuts separated by less than `merge_gap_us`
- if a resulting segment is shorter than `min_segment_duration_us`, merge it into the stronger neighboring
  segment, preferring the neighbor with the larger duration and then the earlier `src_in_us`
- deterministic QC flags are attached before any provider stage runs

Supporting heuristics:

- `blackdetect` for black / near-black coverage
- `freezedetect` for frozen frames
- `signalstats` for luma / highlight statistics
- `silencedetect` and `astats` for silence, clipping, and coarse audio quality

Why this choice:

- deterministic
- already available in the engine layer
- no extra native dependency
- sufficient to replace the fixture segmentation used by M1

Deferred alternative:

- ML shot detection may improve edge cases later, but it adds dependency weight and weaker determinism

#### `segments.json` Base Field Derivation

Per-asset segment shards are emitted after scene detection. Canonical `segments.json` is then written only by
stage reducers; STT and Gemini never patch the canonical file directly.

| `segments.json` field | Producer | Derivation |
| --- | --- | --- |
| `segment_id` | ffmpeg connector | `SEG_<asset_id>_<ordinal>` |
| `asset_id` | ffmpeg connector | copied from asset |
| `src_in_us` | ffmpeg connector | previous cut PTS, or `0` for first segment |
| `src_out_us` | ffmpeg connector | next cut PTS, or asset duration for final segment |
| `duration_us` | ffmpeg connector | `src_out_us - src_in_us` |
| `rep_frame_us` | ffmpeg connector | midpoint frame, adjusted away from black/freeze regions when needed |
| `summary` | ffmpeg connector seeds `""`, Gemini finalizes | fixture-compatible field remains present |
| `transcript_excerpt` | STT connector | overlapping transcript text concatenation |
| `quality_flags` | ffmpeg connector base, Gemini may append | normalized vocabulary only |
| `tags` | ffmpeg connector seeds `[]`, Gemini finalizes | lower_snake_case strings |
| `segment_type` | policy classifier | `static`, `action`, `dialogue`, `music_driven`, or `general` |
| `interest_points` | Gemini connector | array of notable frame/timestamp markers |
| `filmstrip_path` | derivative reducer | deterministic stitched frame strip for MCP preview |
| `waveform_path` | repository convenience path | deterministic segment waveform crop path |
| `confidence.boundary` | ffmpeg connector | normalized cut confidence from scene score / heuristic signal |
| `confidence.summary` / `tags` / `quality_flags` | Gemini or STT | field-level confidence |
| `provenance.*` | all stages | connector, model/method, policy hash, request hash, and run metadata |

Base quality flags from ffmpeg are limited to signals that are physically measurable:

- `black_segment`
- `near_silent`
- `clipped_audio`
- `frozen_frame`
- `underexposed`
- `minor_highlight_clip`
- `very_short_segment`

Subjective or semantic quality flags are deferred to Gemini:

- `out_of_focus`
- `camera_shake`
- `blocked_subject`
- `poor_framing`
- `backlit_face`

#### Contact Sheets

Each analyzed asset generates paginated contact sheets:

- image path: `03_analysis/contact_sheets/CS_<asset_id>_<page>.png`
- manifest path: `03_analysis/contact_sheets/CS_<asset_id>_<page>.json`

Each contact sheet manifest contains:

- `contact_sheet_id`
- `asset_id`
- `image_path`
- `tile_map[]`
  - `tile_index`
  - `segment_id`
  - `rep_frame_us`
  - `src_in_us`
  - `src_out_us`

The manifest exists so `media.open_contact_sheet` can remain unchanged and avoid reparsing image layout at runtime.

#### Posters

Each analyzed asset also generates one deterministic poster image:

- image path: `03_analysis/posters/<asset_id>.jpg`

Poster frame selection rule:

1. rank segments for the asset by:
   - absence of hard reject flags (`black_segment`, `frozen_frame`)
   - longer `duration_us`
   - higher `confidence.boundary.score`
   - earlier `src_in_us`
2. choose the first ranked segment and render `rep_frame_us` to JPEG
3. if every segment is hard-rejected, fall back to the asset midpoint frame

`media.list_assets.poster_path` reads this canonical derivative directly.

#### Filmstrips

Each analyzed segment generates one deterministic filmstrip image:

- image path: `03_analysis/filmstrips/<segment_id>.png`

Filmstrip rule:

- sample 6 evenly spaced frames inside the segment after trimming 5% from each edge
- render each tile at `240 px` width and stitch horizontally in sample order
- if the usable interval is too short for 6 distinct samples, clamp repeated samples to the nearest valid
  frame without changing tile count

`media.peek_segment.filmstrip_path` reads this canonical derivative directly.

#### Waveforms

M2 precomputes one waveform master per asset:

- `03_analysis/waveforms/WF_<asset_id>.png`

Segment-level waveform crops are not required as canonical artifacts. They may be materialized lazily by
`media.peek_segment` or `media.extract_window` using the master waveform plus segment bounds. This keeps M2
cheaper while still satisfying the MCP contract.

### 2. Gemini Video Understanding Connector

Responsibilities:

- enrich `segments.json` with visual semantics
- append model-derived quality flags
- produce concise searchable descriptions
- identify interest points

Input:

- segment visual window defined by `asset_id`, `src_in_us`, `src_out_us`
- sampled frames prepared by the engine layer
- optional transcript excerpt and audio feature summary for context

Output:

- `segments.json` field updates for:
  - `tags`
  - `summary`
  - `interest_points`
  - `quality_flags`
  - `confidence.summary`
  - `confidence.tags`
  - `confidence.quality_flags`
  - `provenance.summary`
  - `provenance.tags`
  - `provenance.quality_flags`

Model assumption:

- provider adapter targets `gemini-2.0-flash`
- canonical runs must persist the resolved `model_snapshot`; alias-only runs are debug-only and not covered by
  byte-stable rerun guarantees

#### Gemini Request Contract

The Gemini connector contract is fixed at the repository boundary even if SDK wiring changes:

- input mode
  - `frame_bundle_plus_text_context`
  - the engine prepares deterministic JPEG frame bundles locally; arbitrary raw clip upload is not canonical
- model identity
  - `model_alias: gemini-2.0-flash`
  - `model_snapshot`: required in canonical provenance and cache keys
- visual limits
  - `max_frames: analysis_policy.vlm.segment_visual_frame_cap` (default `90`)
  - `max_frame_width_px: analysis_policy.vlm.max_frame_width_px` (default `1024`)
- prompt identity
  - `prompt_template_id: m2-segment-v1`
  - `prompt_hash: sha256(normalized_prompt_template + schema_version)`
- expected response schema
  - closed JSON object with `summary`, `tags`, `interest_points`, `quality_flags`, and `confidence`
- parse-retry policy
  - initial parse attempt plus up to `analysis_policy.vlm.parse_retry_max` deterministic repair attempts
  - repair attempts use a separately hashed repair prompt template
- gap fallback
  - if timeout, non-JSON output, or schema-invalid output persists after retries, keep the ffmpeg/STT fields
    already present, leave seeded `summary` / `tags` values untouched, and emit a gap entry

#### Adaptive Sampling Policy

Sampling is policy-driven rather than hardcoded in the connector:

- `static`
  - `0.5 FPS`
  - cap at a small midpoint-biased frame set
- `action`
  - default `4 FPS`
  - allowed range `3-5 FPS`
  - downsample if the token budget would be exceeded
- `dialogue`
  - STT is primary evidence
  - VLM receives sparse supporting frames and transcript context
- `music_driven`
  - VLM is sparse
  - connector prefers beat and waveform evidence from the engine layer over dense frame sampling
- `general`
  - fallback rate between static and action defaults

#### Output Normalization

Gemini output is normalized before writing artifacts:

- `tags`
  - normalized to lower_snake_case
  - deduplicated
  - capped by policy
- `summary`
  - one short sentence
  - descriptive rather than interpretive
- `interest_points`
  - list of `{ frame_us, label, confidence }`
  - must lie within segment bounds
- `quality_flags`
  - mapped to the repository vocabulary; raw provider adjectives are not persisted

#### Cost Control

Each segment obeys a token budget from `analysis_policy.yaml`.

Chosen default:

- `segment_visual_token_budget_max: 8192`
- `segment_visual_output_tokens_max: 512`
- `segment_visual_frame_cap: 90`

If a segment would exceed the budget:

1. reduce sample rate within the policy floor
2. reduce frame width / proxy resolution
3. split the segment into subwindows for analysis and merge the result
4. if still over budget, emit a gap entry instead of silently truncating semantics

Trade-off:

- a hard token budget reduces semantic richness on long or fast segments
- the alternative is unbounded cost and non-repeatable API usage

### 3. OpenAI STT Connector

Responsibilities:

- produce transcript artifacts
- perform diarization
- detect language
- enrich `segments.json` with `transcript_excerpt`
- optionally carry word-level timestamps only when a non-canonical post-alignment pass is enabled

Logical input:

- asset audio is the canonical transcription unit
- long assets are chunked locally before any provider request is made

Chosen orchestration strategy:

- the pipeline transcribes at the asset level first when an audio stream exists
- assets longer than `analysis_policy.stt.chunk_max_us` are pre-split into deterministic audio chunks before
  provider calls
- segment-window transcription is reserved for retries or partial re-analysis after the canonical asset pass

Why asset-level first:

- diarization is more stable across a whole source file than across isolated segments
- speaker keys stay consistent
- fewer API round-trips

Model assumption:

- endpoint: `POST /v1/audio/transcriptions`
- model: `gpt-4o-transcribe-diarize`
- response format: `diarized_json`
- canonical request path uses client-side chunking and keeps each uploaded chunk at or below
  `analysis_policy.stt.chunk_max_us`; provider-side `chunking_strategy` is therefore `none` in canonical runs
- provider-side `chunking_strategy: auto` is permitted only for manual debugging and is excluded from
  byte-stable rerun guarantees

#### Chunking And Merge Rules

Canonical chunking policy:

- audio proxy: mono `16 kHz` PCM WAV extracted from the asset source
- `analysis_policy.stt.chunk_target_us: 20000000`
- `analysis_policy.stt.chunk_max_us: 25000000`
- `analysis_policy.stt.chunk_overlap_us: 500000`
- `analysis_policy.stt.chunk_boundary_silence_us: 350000`
- chunk boundaries are chosen by silence-aware splitting first; if no silence boundary is available before
  `chunk_max_us`, hard-cut at `chunk_target_us` and apply the configured overlap

Chunk merge rules:

1. convert chunk-relative diarized segment times to asset-level `start_us` / `end_us`
2. stable-sort merged utterances by `start_us`, `end_us`, and normalized text
3. treat overlap-region items as duplicates when normalized text matches and temporal overlap is at least `80%`
4. keep the earliest duplicate and drop later copies created by chunk overlap
5. do not semantic-merge non-identical utterances across chunks; the transcript artifact preserves provider
   segmentation after duplicate removal

#### Naming Rules

- `transcript_ref`
  - `TR_<asset_id>`
- transcript item `speaker`
  - fixture-compatible short label such as `S1`, `S2`
- transcript item `speaker_key`
  - `<asset_id>:speaker_<n>`
- transcript item `item_id`
  - `TRI_<asset_id>_<ordinal_4>`

#### Speaker Label Normalization

Provider speaker labels are treated as chunk-local hints. Canonical labels are normalized at the asset level:

- first speaker encountered after merged sort order becomes `S1`, next distinct speaker becomes `S2`, and so on
- overlap duplicates across adjacent chunks anchor speaker identity when the same utterance appears twice
- if a new chunk speaker cannot be matched through overlap anchoring, it receives the next canonical speaker id
- `speaker_key` is always `<asset_id>:speaker_<n>`

#### Transcript Alignment

The transcript artifact is asset-scoped:

- path: `03_analysis/transcripts/TR_<asset_id>.json`
- contains ordered diarized utterance items; `words[]` is optional and controlled by `word_timing_mode`

`segments.json.transcript_excerpt` is derived after the transcript is written:

- include transcript items whose overlap with the segment is at least `250 ms` or `25%` of the item duration
- concatenate overlapping text in source order
- do not store speaker labels in the excerpt; keep the fixture-friendly plain-text field

#### Word-Level Timestamps

`gpt-4o-transcribe-diarize` in canonical M2 mode supplies diarized utterance timing, not word arrays.

M2 decision:

- `words[]` is not required for the canonical live-ready transcript profile
- transcript excerpt alignment uses utterance `start_us` / `end_us`, not `words[]`
- if a future post-alignment substage is enabled, it may add `words[]` with separate provenance, but that path
  is explicitly outside canonical M2 determinism guarantees
- if diarization fails for a chunk, the stage records a gap and leaves the asset transcript in `partial` or
  `failed` state instead of fabricating speaker labels

#### STT Provider Migration Path

M2 uses OpenAI `gpt-4o-transcribe-diarize` for simplicity: diarization is built-in, reconciliation logic is
unnecessary, and the pipeline validation can focus on artifact flow rather than STT optimization.

However, the transcript schema is designed to accommodate richer STT output for future provider migration:

- `words[]` is optional and controlled by `word_timing_mode`
- `confidence` fields accept per-utterance and per-word scores
- `provenance` tracks model, snapshot, and request metadata

Target migration (M3 or later): **Groq Whisper (`whisper-large-v3`) + pyannote.audio** for:

- word-level timestamps with `avg_logprob` and `no_speech_prob` (editing-grade intermediate representation)
- lower cost (~$0.002/min vs ~$0.006/min) at higher throughput (~200x speed factor)
- fine-grained confidence for low-reliability segment re-analysis and prompt-based proper noun correction
- self-hosted diarization with tunable DER thresholds

The STT connector interface is provider-agnostic: `analysis_policy.stt.model_alias` selects the backend.
Adding a Groq + pyannote backend requires implementing the ASR-diarization reconciliation logic
(speaker cluster alignment, word-utterance merge, cross-chunk speaker identity) but does not change
downstream artifact shapes or `media-mcp` contracts.

## Analysis Policy

Milestone 2 introduces a policy file with two layers:

- repo default: `runtime/analysis-defaults.yaml`
- optional project override: `projects/<project_id>/analysis_policy.yaml`

Merge order:

1. repo defaults
2. project override
3. explicit runtime flags for a one-off job

The merged result is persisted by hash in artifact provenance so re-runs can prove idempotency.

Schema contract:

- `schemas/analysis-policy.schema.json` is a Phase 1 deliverable
- the merged policy is hashed only after schema validation and canonical key sorting
- canonical root shape is closed (`additionalProperties: false`)

#### Classification Rules

Segment type is chosen after base ffmpeg analysis and, when available, after STT:

- `dialogue`
  - audio stream present
  - speech ratio above threshold or transcript words-per-second above threshold
- `music_driven`
  - speech ratio below dialogue threshold
  - beat strength above threshold
  - audio energy variance above threshold
- `action`
  - motion score above threshold or cut density above threshold
- `static`
  - motion score low
  - no strong beat or dialogue evidence
- `general`
  - fallback when no specific class wins

Default rule ordering:

1. dialogue
2. music_driven
3. action
4. static
5. general

This ordering is intentional: dialogue and music-driven segments need different downstream budgets and should
override pure motion heuristics.

#### VLM Call Conditions

Gemini is called when:

- segment duration is above the minimum duration threshold
- the segment is not already excluded by deterministic flags such as `black_segment`
- the segment has not already been enriched with the same policy hash and model snapshot

Gemini is skipped when:

- the segment is marked unusable by hard flags
- STT-only dialogue segments already satisfy the policy's minimum evidence requirement
- a previous successful enrichment with matching cache key exists

#### STT Call Conditions

OpenAI STT is called when:

- an audio stream exists
- duration exceeds the minimum speech-analysis threshold
- the asset is not already cached with the same policy hash and model snapshot

STT is skipped when:

- no audio stream exists
- audio is near-silent for the entire asset and policy allows omission
- a prior artifact already matches the same cache key

#### Quality Thresholds

The policy also carries deterministic thresholds such as:

- black coverage ratio
- silence ratio
- clipping threshold
- minimum segment duration
- beat strength threshold
- transcript overlap threshold for excerpt extraction

These thresholds must be machine-readable and not hidden in connector code. That closes the M1 review gap
about untracked analysis policy.

## Pipeline Orchestration

### Stage Order

The canonical M2 execution order is:

1. `ingest.map`
   - run `ffprobe` per asset
   - write per-asset ingest shards
2. `ingest.reduce`
   - write canonical `assets.json` through temp file + atomic rename
3. `segment.map`
   - run scene detection and deterministic QC per asset
   - write per-asset segment shards
4. `segment.reduce`
   - write canonical `segments.json` through temp file + atomic rename
5. `derivatives.map`
   - render contact sheets, posters, filmstrips, and waveform masters
   - write derivative shards only
6. `derivatives.reduce`
   - rewrite canonical `assets.json` and `segments.json` with derivative refs
7. `stt.map`
   - transcribe asset audio chunks
   - write transcript shards and transcript-excerpt shards
8. `stt.reduce`
   - write `transcripts/TR_*.json`
   - rewrite canonical `assets.json` and `segments.json` with transcript refs and excerpts
9. `vlm.map`
   - enrich segments with Gemini
   - write per-segment enrichment shards
10. `vlm.reduce`
    - rewrite canonical `segments.json` with summaries, tags, interest points, and model quality flags
11. `gap.reduce`
    - union stage gap shards into canonical `gap_report.yaml` through temp file + atomic rename

Canonical write discipline:

- `assets.json`, `segments.json`, and `gap_report.yaml` are written only by reducers
- map workers never patch canonical artifacts directly
- reducers consume the previous canonical snapshot plus stage-local shards, then replace the target file by
  atomic rename in the same directory

### Parallelism

Safe parallelization units:

- per asset:
  - `ingest.map`
  - `segment.map`
  - `derivatives.map`
  - `stt.map`
- per segment:
  - `vlm.map` after `stt.reduce` and `segment.reduce` are complete

Single-writer stages:

- `ingest.reduce`
- `segment.reduce`
- `derivatives.reduce`
- `stt.reduce`
- `vlm.reduce`
- `gap.reduce`

Concurrency must be policy-controlled:

- ffmpeg jobs: CPU-bound, local
- STT jobs: provider-limited
- VLM jobs: provider-limited and cost-sensitive

### Partial Failure Handling

M2 must preserve partial results.

Operational artifact:

- `03_analysis/gap_report.yaml`
- `03_analysis/run_manifest.json`

Each gap entry contains:

- `stage`
- `asset_id`
- optional `segment_id`
- `severity`
- `blocking`
- `retriable`
- `reason`
- `attempted_at`

Examples:

- STT failed for one asset but segments are still available
- Gemini enrichment timed out for several segments
- contact sheet image write failed after `segments.json` already succeeded

`media.project_summary` aggregation rules:

- `assets_total`: count of items in `assets.json`
- `segments_total`: count of items in `segments.json`
- `transcripts_available`: `true` if at least one asset has `has_transcript: true`
- `contact_sheets_available`: `true` if at least one `contact_sheet_ids` entry exists
- `top_motifs`: collect all `tags` from `segments.json`, count occurrences, return the top 5 by frequency;
  tie-break by alphabetical order
- `analysis_gaps`: projection of unresolved `gap_report.yaml` entries (see below)
- `qc_status`: derived from asset-level analysis status (see below)

`media.project_summary.analysis_gaps` is a projection of unresolved `gap_report.yaml` entries:

- sort by `blocking desc`, `severity desc`, `stage asc`, `asset_id asc`, `segment_id asc`
- render each entry as `"<severity>/<stage>/<asset_id>[/<segment_id>]: <reason>"`
- preserve the full sorted set; the MCP layer does not invent aggregate prose

`media.project_summary.qc_status` derives from this state:

- `ready`
  - no blocking gaps
- `partial`
  - at least one non-blocking gap remains
- `blocked`
  - ingest or segmentation failed such that the asset set is unusable for analysis

Asset-to-project crosswalk:

| Asset-level state mix | Project `qc_status` |
| --- | --- |
| all assets `ready` or intentionally `skipped`, no blocking gaps | `ready` |
| at least one asset `ready`, and any mix of `partial`, `failed`, or `skipped` with only non-blocking gaps | `partial` |
| no asset `ready`, and at least one ingest/segment blocking failure | `blocked` |
| dialogue assets missing required transcript coverage under `gap_policy.block_on_missing_transcript_for_dialogue` | `blocked` |
| mixed `ready` + `failed` assets where failed assets are non-blocking for the project goal | `partial` |

Retry policy:

- retries are stage-specific and bounded by `gap_policy.max_retries`
- retry exhaustion must create or update a gap entry; it may not silently downgrade the stage to success

### Idempotency

Re-running the pipeline with the same inputs must yield the same artifacts.

The cache key for each stage is:

- source fingerprint
- connector version
- policy hash
- model snapshot
- prompt hash
- response format
- chunking strategy
- ffmpeg version
- stage-specific request hash

Determinism is defined by the same canonical request tuple, not by cache reuse alone. Cache reuse is an
optimization; reducers must still be able to rebuild the same byte-stable artifacts from scratch.

Idempotency rules:

- artifacts are written in stable sort order
- JSON serialization is normalized
- derivative paths are deterministic from stable IDs
- successful stage outputs are reused on re-run when the cache key matches
- partial re-run is allowed at stage granularity without renumbering unaffected assets

## `media-mcp` Integration

Milestone 1 fixed the tool contract in
[contracts/media-mcp.md](/Users/mocchalera/Dev/video-os-v2-spec/contracts/media-mcp.md). Milestone 2 must not
change request or response shapes.

Chosen integration pattern:

- tool interface unchanged
- backend selected by factory + config injection

TypeScript-level design:

- `AnalysisRepository`
  - read-only interface used by MCP tools
  - returns assets, segments, transcripts, contact sheet manifests, and gaps
- `FixtureAnalysisRepository`
  - current M1 mode reading static fixture files
- `LiveAnalysisRepository`
  - reads the same artifact paths after the live pipeline populates them
- `ConnectorFactory`
  - creates ffmpeg, Gemini, and STT adapters from runtime config
- `AnalysisPipeline`
  - orchestrates stage execution and writes artifacts before the MCP layer reads them

Why factory + injection:

- preserves the M1 tool interface
- allows tests to swap fixture, mock, and live backends cleanly
- keeps provider-specific SDK code out of the MCP tool layer

Additional M2 behavior without contract changes:

- `media.search_segments` uses lexical/hybrid search over `summary`, `tags`, and `transcript_excerpt`
  until embeddings are introduced later
- `media.open_contact_sheet` reads the generated contact sheet manifest
- `media.list_assets.poster_path` reads canonical `assets.json.items[].poster_path`
- `media.peek_segment.filmstrip_path` reads canonical `segments.json.items[].filmstrip_path`
- `media.peek_segment.waveform_path` resolves a deterministic crop path
  `03_analysis/waveforms/<segment_id>.png`; the crop may be materialized lazily from the asset waveform
  master, but lazy materialization may not mutate canonical JSON
- `media.project_summary.analysis_gaps` is projected deterministically from `gap_report.yaml`

## Trade-offs And Deferred Work

### Chosen For M2

- ffmpeg scene detection over ML shot detection
- asset-level STT first, segment-level retry only
- hash-stable asset IDs
- asset-level waveform master with lazy segment crop
- no embeddings dependency yet

### Deferred Beyond M2

- embeddings-backed semantic retrieval
- richer beat-alignment analysis for music-driven segments
- project-level asset identity registry for cross-project deduplication
- full search ranking that mixes embeddings, transcript, and VLM evidence
- automatic `selects_candidates.yaml` generation by product-plane agents

### Rollback Strategy

If live analysis introduces instability, the repository must be able to fall back without changing the MCP
surface or downstream compiler inputs:

- keep `FixtureAnalysisRepository` available behind the same factory
- gate live backend selection behind config instead of hard-switching the tool layer
- treat `03_analysis/` as regenerable outputs; rollback means reselecting the backend and regenerating, not
  mutating downstream `04_plan/` or `05_timeline/` contracts
- if a provider integration becomes unavailable, preserve ffmpeg-only artifact generation and surface the API
  gap explicitly rather than silently reverting to fixture-like invented data

## Final Design Notes

This design keeps M2 narrow: it turns `03_analysis/` from fixture data into live-generated evidence without
changing the editorial loop contract that M1 already validated.

The key constraint is backward compatibility:

- old fixtures must keep validating
- new live artifacts must carry enough provenance and confidence for M3 agents
- `media-mcp` must not expose provider churn or implementation details upward

That makes M2 the analysis bridge, not a redesign of the rest of the system.
